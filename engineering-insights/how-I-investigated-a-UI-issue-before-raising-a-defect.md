# How I Investigated a UI Issue Before Raising a Defect

## The Question

During the AC1 investigation, the home page displayed products in a table/list-style layout instead of the expected grid of product cards.

Rather than immediately raising a defect, I investigated whether another explanation could account for the observed behaviour.

The goal was not to prove that a bug existed, but to systematically rule out alternative explanations before reaching that conclusion.

---

## Step 1 – Can I Reproduce the Behaviour?

Before investigating further, I confirmed that the behaviour was reproducible.

I repeated the same steps and observed the same result.

I also repeated the investigation in Firefox.

The behaviour remained unchanged.

### What this ruled out

- Temporary application behaviour.
- Browser-specific rendering issues.

---

## Step 2 – Could This Be Responsive Behaviour?

Some websites intentionally display different layouts depending on the screen size.

I confirmed that the investigation was performed using a desktop viewport.

### What this ruled out

- Mobile or tablet responsive layouts.

---

## Step 3 – Check the Console

The next question was:

> Did part of the application fail while loading?

I opened the Console and looked for JavaScript errors.

Examples include:

Uncaught TypeError...

or

ReferenceError...


If the JavaScript responsible for rendering the page crashes, the application may display only part of the intended interface.

I also looked for failed resource messages such as:

Failed to load resource

### What this could have explained

- Broken page rendering.
- Missing functionality.
- Incomplete user interface.

### What I found

No obvious JavaScript errors explained the observed behaviour.

### What this ruled out

The layout was unlikely to be caused by an obvious frontend runtime failure.

---

## Step 4 – Check the Network

Next I asked:

> Did every required resource load successfully?

The Network tab helped answer that question.

Every request made by the browser appears here, allowing me to verify whether the page received all of the resources and data it expected.

I checked:

### Product API requests

If these had failed, I might have expected:

- missing products;
- incomplete data;
- loading indicators;
- HTTP errors such as 404 or 500.

Products loaded successfully.

---

### JavaScript files

A failed JavaScript download could explain an incomplete page.

No failed JavaScript requests were observed.

---

### CSS files

If the stylesheet responsible for the grid failed to load, products might still appear but without the intended layout.

No failed stylesheet requests were observed.

---

### What this ruled out

The observed behaviour was unlikely to be caused by:

- missing product data;
- failed JavaScript downloads;
- failed stylesheet downloads;
- obvious frontend loading problems.

---

## Step 5 – Inspect the Elements

This turned out to be the most important part of the investigation.

The question was no longer:

> Did something fail?

Instead, it became:

> What did the application actually render?

Using the Elements panel, I inspected the HTML generated by the application.

If I had found a grid structure in the HTML but the page still looked like a table, I would have suspected a CSS problem.

Instead, the HTML itself reflected the table/list-style structure.

This was a significant finding.

The browser was not failing to render a grid.

The application had actually generated a table/list-style structure.

That shifted the investigation away from CSS or rendering problems and towards a possible implementation or requirement mismatch.

### What this suggested

The application was intentionally rendering this layout.

The issue was therefore less likely to be caused by missing CSS or failed frontend resources.

Instead, it suggested one of two possibilities:

- the implementation differs from the documented acceptance criteria;
- the acceptance criteria are no longer up to date.

This observation became the strongest piece of evidence in the investigation.

---

## Final Assessment

By the end of the investigation, I had ruled out:

- browser compatibility;
- responsive layout;
- JavaScript runtime failures;
- failed resource loading;
- missing product data;
- obvious stylesheet loading failures.

Other explanations were still possible, including:

- implementation that does not match the documented requirement;
- outdated acceptance criteria;
- a feature flag or configuration that changes behaviour for specific users;
- a deployment issue;
- a difference between environments.

At that point, further clarification was required before deciding whether to raise a defect.

---

## Key Takeaway

A QA investigation should not begin by asking:

> "Is this a bug?"

Instead, it should begin by asking:

> "What else could explain what I'm seeing?"

Only after systematically ruling out alternative explanations should a potential defect be reported.

Good QA investigation is a process of eliminating possibilities.

The goal is not to prove that a defect exists, but to gather enough evidence to confidently explain what is happening and why.
